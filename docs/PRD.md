## 1. 背景与问题

公司内部已有并将持续新增一批自研应用（客服中枢 CS Hub、各类内部工具、后台系统等）。当前每个应用**各自维护一套账户体系**，带来四类结构性问题：

1. **入职开通靠人肉**：新员工要用某系统，得管理员手工建号/授权，慢、易漏、无审计。
2. **离职停用不闭环**（最大风险）：员工离职后，各系统的账号靠人工逐个关，**漏关一个就是一个长期敞口**——前员工仍能访问内部数据。
3. **无统一登录**：每个应用一套登录，员工记多套密码，弱密码/复用普遍；没有 SSO。
4. **无统一审计与授权模型**：谁有哪个系统的什么权限，散落各处，说不清、审不动。

同时有三条硬约束：**自托管**（身份数据必须留在阿里云内网，不能用纯 SaaS）、**千人级**规模、**将来要接多身份源**（当前只有飞书，未来邮箱/密码、其他 OIDC/SAML、外部合作方）。

**结论**：需要一个统一账户管理平台，把"一个身份、一次登录、一套生命周期"沉淀成公司基础设施，供所有内部应用共享。

## 2. 产品愿景与目标

### 2.1 愿景

> **一个身份，一次登录，一套生命周期，横跨所有内部应用。**
> 员工用飞书（及未来其他身份源）一次认证即可访问所有获授权的内部应用；入职经审批自动开通、离职随飞书账户自动全线停用；每个应用的准入与角色集中治理、可审计。

### 2.2 目标（可量化）

| 目标 | 指标 |
|---|---|
| 统一登录 | 所有新接入应用走标准 OIDC，员工单点登录，不再各自维护密码 |
| 离职即断权 | 离职/冻结后 **≤5 分钟**，员工在所有应用的访问被切断 |
| 入职提效 | 准入从"人肉建号（小时/天级）"降到"扫码 + 审批人一键批准（分钟级）” |
| 接入低成本 | 新应用接入 ≤1 人日（标准 OIDC + 抄示例 + 建准入组） |
| 可审计 | 每一次准入/拒绝/停用/角色变更均留痕可查 |
| 自托管合规 | 身份数据 100% 留在阿里云内网 |

### 2.3 非目标（本平台不做）

- 不做**应用内业务权限的语义**（"某角色能点哪个按钮"由各应用自己定义，平台只下发角色名）。
- 不做通用 HR 系统 / 考勤 / 组织架构主数据（以飞书为组织真理源）。
- 不做面向外部终端用户（ToC）的账号体系（本平台面向内部员工 + 受控外部合作方）。

## 3. 目标用户与角色

| 角色 | 诉求 |
|---|---|
| **员工**（终端用户） | 一次飞书登录访问所有获授权应用；离职后自动收权，无需操心 |
| **审批人 / 主管** | 收到飞书审批卡片，一键批准/拒绝下属对某应用的准入；可按应用不同人审批 |
| **IT / 平台管理员** | 管身份源、应用注册、组/角色、审计；处理停用失败告警；备份恢复 |
| **应用开发者** | 用标准 OIDC 低成本接入，读 claim 做准入/角色判定，不碰飞书密钥 |
| **安全 / 审计** | 查任意时刻"谁能访问什么"；核对离职撤权闭环；导出审计轨迹 |

## 4. 核心概念与术语

| 术语 | 含义 |
|---|---|
| **IdP / Casdoor** | 身份提供方。平台以开源 Casdoor 为协议底座，承接飞书原生登录、OIDC/SAML、会话、多应用、管理后台。身份真理源。 |
| **治理服务 even-auth-gov** | 平台自研控制器。负责入职审批工作流、离职三层闭环、飞书交互（审批卡片/事件），是把"账户生命周期治理"沉淀成产品的核心。 |
| **open_id / sub** | 飞书用户唯一标识，= OIDC id_token 的 `sub`，各应用的用户主键。 |
| **准入（groups claim）** | 用户能否进某应用。以 `approved-<app>` 组表达，进 id_token `groups` claim。 |
| **角色（roles claim）** | 用户在某应用内的角色。以 `<app>-<role>` 表达，进 `roles` claim。 |
| **认证 / 授权分离** | Casdoor 答"你是谁"，准入组/角色答"你能用什么、能干什么"。 |

## 6. 功能需求

### 6.1 身份认证

- **F-6.1.1** 支持**飞书扫码登录**（原生 provider，员工无需额外账号）。首次登录 Casdoor 自动建号，`sub` = 飞书 open_id。
- **F-6.1.2**（Roadmap）支持**多身份源**：邮箱/密码、其他 OIDC/SAML IdP、受控外部合作方账号。平台以标准协议为底座，加身份源不改应用。
- **F-6.1.3** 认证入口唯一化：应用切到平台后，关闭其自建的直连登录（避免绕过平台的第二授权源）。

### 6.2 单点登录（SSO）

- **F-6.2.1** 应用以**标准 OIDC 客户端**接入，员工一次登录后免登进入所有获授权应用（父域会话 / Casdoor SSO 会话）。
- **F-6.2.2** 支持 OIDC 与 SAML 两类协议，覆盖不同技术栈的应用。
- **F-6.2.3** 应用**不持有任何飞书密钥**，只信任平台。

### 6.3 入职准入与审批

- **F-6.3.1 每应用独立准入**：准入按 **(用户, 应用)** 记录，同一用户可对应用 A 待批、对应用 B 已批，互不影响。
- **F-6.3.2 审批工作流**：用户首次访问某应用且未获准入 → 应用展示"待审批"页并向平台上报（`POST /request-access`）→ 平台建待批记录 + 推**飞书交互审批卡片**给该应用的审批人。
- **F-6.3.3 每应用审批人**：审批卡片按应用路由到不同审批人（`APPROVER_<APP>`），CS Hub 的申请给客服主管、财务系统给财务负责人。
- **F-6.3.4 一键批准/拒绝**：审批人在飞书卡片上点批准 → 平台把用户加入 `approved-<app>` 组 → 用户重登即放行；拒绝则撤组 + 记录。
- **F-6.3.5 审批可靠性**：卡片发送失败/审批人误配 → 待批不永久悬挂，对账任务补发；被拒用户可重新申请；已批准/已停用用户的重复事件不重置状态。

### 6.4 离职与账户停用（平台核心价值）

- **F-6.4.1 事件驱动停用**：订阅飞书 `contact.user.deleted_v3 / updated_v3`，员工离职/冻结/离职 → 实时禁用其 Casdoor 账号（一次性关闭全部应用）。
- **F-6.4.2 每日对账兜底**：定时任务核对"平台记为有权"的用户在飞书的在职状态；漏掉的事件由对账补上。对账检查集 = 本地审批库 ∪ Casdoor 准入组成员（迁移/手工授权用户也覆盖）。
- **F-6.4.3 硬删除识别**：飞书查无此人（硬删除）判定为离职并停用，与"飞书 API 瞬时不可用"区分（后者 fail-open 不误禁）。
- **F-6.4.4 停用绝不遗漏**：禁用失败 → 退避重试 → 仍失败则标记 + 飞书告警管理员人工介入，对账每日重扫失败项。
- **F-6.4.5 分钟级全线撤权**：应用会话短时效 + 定期回平台复验（禁用用户复验被拒即踢出），保证离职后 ≤5 分钟所有应用断权。

### 6.5 授权模型（三层）

| 层 | 管什么 | 机制 | claim |
|---|---|---|---|
| **① 认证** | 你是谁 | Casdoor + 飞书 | `sub` = open_id |
| **② 准入** | 能不能进应用 X | 每应用组 `approved-<app>` | `groups` |
| **③ 角色权限** | 进去能干什么 | 每应用角色 `<app>-<role>` | `roles` |

- **F-6.5.1 分配中心化**：谁有什么准入/角色，集中在平台治理，一处审计、离职一键全线失效。
- **F-6.5.2 含义应用自理**：某角色能干什么由应用定义，平台只下发角色名。加应用零平台负担。
- **F-6.5.3 审批即赋角**（可选简化）：审批时可直接选角色，"有该应用任一角色 = 准入，具体角色 = 权限"。

### 6.6 会话与令牌撤销

- **F-6.6.1** 应用会话短时效 + 服务端会话表 + 定期回 Casdoor 复验（refresh_token grant）。
- **F-6.6.2** 登出即服务端销毁会话；离职即复验失败踢出。
- **F-6.6.3** IdP 不可达时有硬上限的 fail-open（不误杀全员，但离职者不能靠 IdP 宕机续命）。

### 6.7 应用接入

- **F-6.7.1** 提供标准 OIDC 接入规范 + 可跑的最小示例（≤100 行客户端）。见 `docs/integration-guide.md`。
- **F-6.7.2** 接入方三件事：接标准 OIDC、查准入组 claim、待批时上报平台。飞书/审批/离职撤权全由平台兜住。
- **F-6.7.3 安全须知强制项**：id_token 裁敏感字段（`tokenFormat=JWT-Custom` + 白名单）、生产 JWKS 验签、回调 HTTPS。

### 6.8 管理后台与审计

- **F-6.8.1** Casdoor 管理后台：身份源、应用、组、角色、会话管理。
- **F-6.8.2** 审计轨迹：准入/拒绝/停用/角色变更/停用失败均留痕，可查询导出。
- **F-6.8.3**（Roadmap）平台自建的"一张表看清谁能访问什么"视图 + 离职撤权核对报表。

### 6.9 通知

- **F-6.9.1** 审批卡片、停用失败告警、（Roadmap）申请结果回告申请人，均走飞书。

## 7. 非功能需求

| 维度 | 要求 |
|---|---|
| **安全** | 认证/授权分离；准入 fail-closed；离职绝不漏禁；密钥入密管；id_token 裁敏感字段；审批回调仅走验签通道 |
| **可用性** | Casdoor + 治理服务 `restart: unless-stopped`；治理服务**单实例**（WS + 文件状态 + 进程内锁），高可用用主备拉起 |
| **性能** | 千人级规模；登录/复验低延迟；对账任务离峰跑 |
| **可扩展** | 加身份源不改应用；加应用不改平台（标准协议 + 每应用组/角色） |
| **合规审计** | 身份数据留阿里云内网；全流程留痕；备份可恢复 |
| **部署运维** | Docker Compose + .env 驱动；nginx 反代 TLS；Casdoor 库每日 pg_dump 备份 + 异地；治理服务日志可采集 |

## 8. 系统架构

- **Casdoor**（身份底座）：飞书 provider、OIDC/SAML 端点、会话、组/角色、管理后台。Postgres 为唯一身份真理源。
- **even-auth-gov**（治理服务，平台核心）：飞书 WS 长连接（离职事件 + 审批卡片回调）、`/casdoor/webhook`（新用户）、`/request-access`（应用上报待批）、离职引擎、对账任务、审批状态机、Casdoor Admin API 封装。**自带飞书集成，不依赖任何业务应用。**
- **应用（OIDC 客户端）**：CS Hub 等，读 id_token 的 groups/roles 做准入与角色判定。

数据流：飞书扫码 → Casdoor 认证（sub=open_id）→ 应用 OIDC → 查 groups 准入 → 治理服务审批加组 → 放行；离职 → 飞书事件/对账 → 治理服务禁用 Casdoor 账号 → 各应用复验断权。

---

## 9. 数据模型（概览）

- **身份**（Casdoor）：用户（sub/open_id、name、email、groups、roles、isForbidden）。
- **组 / 角色**（Casdoor）：`approved-<app>` 准入组、`<app>-<role>` 角色。
- **审批状态**（治理服务，JSON 文件，按 (open_id, app)）：pending / approved / denied / disabled / disable_failed + 审计字段（申请/批准/拒绝/停用时间与操作人、通知留痕）。
- **会话**（应用侧服务端会话表）：sid → sub / refresh_token / roles / last_validated。

> 审批/会话状态**不入随每日 dump 外泄的库**，单独文件存储。

## 11. 现状与差距（诚实盘点）

**已建成并验证**：
- 飞书扫码登录 → Casdoor → 应用 OIDC 的完整链路（真环境实证）。
- 入职审批闭环（含飞书卡片按钮点击 → 加组 → 放行，真机跑通）。
- 离职即断权闭环（事件 + 对账 + 硬删除识别 + 会话回查 + 停用重试告警）。
- 多应用授权模型（每应用准入组/角色/审批人 + `/request-access`）。
- 部署加固（.env 驱动、备份脚本、单实例约束文档、治理服务日志）。

**待办（非代码欠账，平台侧配置 / 后续阶段）**：
1. **平台配置**：飞书应用发正式版（消息权限 + 事件订阅）、Casdoor 每应用配 webhook + 裁 token 字段、生产密钥入密管、nginx TLS、备份 cron 落地。
2. **RBAC 强制**：角色已进会话（地基就位），"哪些路由按哪个角色拦"待定义角色模型后开启。
3. **多身份源 / 自建管理视图 / SCIM**：Phase 3–4 规划。

## 12. 成功指标（KPI）

- 离职后全线断权时延 **≤5 分钟**（抽样核对）。
- 漏停率 **0**（离职员工在任何应用留有有效访问 = 事故）。
- 新应用接入工时 **≤1 人日**。
- 接入应用数从 1（CS Hub 试点）增长到 6–15。
- 审批 SLA：申请到批准中位时长（目标 < 1 工作日）。

## 14. 范围外

- 应用内业务权限的具体语义。
- 组织架构 / HR 主数据（以飞书为准）。
- 面向外部终端用户（ToC）的账号体系。
- 多活高可用（当前单实例；列为将来改造）。
