# 企业内部统一账户管理平台 — 产品需求文档（PRD）

| 项 | 值 |
|---|---|
| 文档名 | 企业内部统一账户管理平台 — 产品需求文档 |
| 版本 | v1.0 |
| 状态 | 草案 |
| 适用范围 | 内部 6–15 个自研应用、千人级员工规模、自托管（内网） |

> 本文只描述**要什么、为什么**（产品需求），不涉及技术选型、系统实现与开发进度——换实现不改本文。方案设计见另行的设计文档。

---

## 1. 背景与问题

公司内部持续新增自研应用，每个应用**各自维护一套账户体系**，带来四类结构性问题：

1. **入职开通靠人肉**：新员工要用某系统，需管理员手工建号/授权——慢、易漏、无审计。
2. **离职停用不闭环（最大风险）**：员工离职后各系统账号靠人工逐个关，**漏关一个就是一个长期敞口**，前员工仍能访问内部数据。
3. **无统一登录**：每个应用一套登录，员工记多套密码，弱密码与复用普遍，没有 SSO。
4. **无统一授权与审计**：谁有哪个系统的什么权限散落各处，说不清、审不动。

**硬约束**：
- **自托管**：身份数据必须留在公司内网，不能用纯 SaaS。
- **千人级**员工规模。
- **多身份源可扩展**：当前身份源是飞书，未来须能接入邮箱/密码、其他标准身份提供方（OIDC/SAML）、受控外部合作方。

---

## 2. 产品愿景与目标

### 2.1 愿景

> **一个身份、一次登录、一套生命周期，横跨所有内部应用。**
> 员工用一次认证即可访问所有获授权的内部应用；入职经审批自动开通、离职随身份源账户自动全线停用；每个应用的准入与角色集中治理、可审计。

### 2.2 目标（可量化）

| 目标 | 指标 |
|---|---|
| 统一登录 | 所有新接入应用走统一 SSO，员工单点登录，不再各自维护密码 |
| 离职即断权 | 离职/冻结后 **≤5 分钟**，员工在所有应用的访问被切断 |
| 入职提效 | 准入从"人肉建号（小时/天级）"降到"申请 + 审批人一键批准（分钟级）" |
| 接入低成本 | 新应用接入 **≤1 人日** |
| 可审计 | 每一次准入/拒绝/停用/角色变更均留痕，可查询导出 |
| 自托管合规 | 身份数据 **100% 留在公司内网** |

### 2.3 非目标（本平台不做）

- 不做**应用内业务权限的语义**（"某角色能点哪个按钮"由各应用自定义，平台只声明角色）。
- 不做通用 HR 系统 / 考勤 / 组织架构主数据（组织与在职状态以既有身份源为准）。
- 不做面向外部终端用户（ToC）的账号体系（本平台面向内部员工 + 受控外部合作方）。

---

## 3. 目标用户与角色

| 角色 | 诉求 |
|---|---|
| **员工**（终端用户） | 一次登录访问所有获授权应用；离职后自动收权，无需操心。 |
| **审批人 / 主管** | 收到申请通知，一键批准/拒绝下属对某应用的准入；不同应用可由不同人审批。 |
| **平台管理员** | 管身份源、应用、准入与角色、审计；处理停用异常；保障可恢复。 |
| **应用开发者** | 低成本接入统一登录，读取身份与授权信息做判定，无需自建账户体系。 |
| **安全 / 审计** | 查任意时刻"谁能访问什么"；核对离职撤权闭环；导出审计轨迹。 |

---

## 4. 关键概念

| 概念 | 含义（产品视角） |
|---|---|
| **身份源** | 提供员工身份的上游系统（当前为飞书，未来可多个）。 |
| **单点登录（SSO）** | 员工认证一次，即可免登进入所有获授权应用。 |
| **准入** | 某员工能否进入某个应用（应用级、逐个授予/回收）。 |
| **角色** | 某员工在某应用内的角色（应用级；角色的具体权限由应用自定义）。 |
| **入职闭环** | 员工首次访问应用 → 审批 → 开通准入。 |
| **离职闭环** | 员工离职/冻结 → 自动停用其在所有应用的访问。 |

## 5. 功能需求

### 5.1 身份认证
- **F-5.1.1** 支持员工用**身份源账号登录**（当前飞书；首次登录自动建立平台身份）。
- **F-5.1.2** 支持**多身份源**：邮箱/密码、其他标准身份提供方、受控外部合作方；加身份源不改应用。
- **F-5.1.3** **认证入口唯一化**：应用接入平台后，须关闭其自建的直连登录，避免绕过平台的第二授权源。

### 5.2 单点登录（SSO）
- **F-5.2.1** 员工一次登录后**免登**进入所有获授权应用。
- **F-5.2.2** 支持业界标准 SSO 协议，覆盖不同技术栈的应用互通。
- **F-5.2.3** 应用**不持有身份源密钥**，只信任平台。

### 5.3 入职准入与审批
- **F-5.3.1 每应用独立准入**：准入按 **(员工, 应用)** 记录，同一员工可对应用 A 待批、对应用 B 已批，互不影响。
- **F-5.3.2 审批工作流**：员工首次访问某应用且未获准入 → 展示"待审批"状态并进入审批流。
- **F-5.3.3 每应用审批人**：审批请求路由到该应用对应的审批人；不同应用可由不同人审批。
- **F-5.3.4 一键批准/拒绝**：审批人一键批准即开通准入、拒绝即记录；结果对申请人可见。
- **F-5.3.5 审批可靠性**：通知发送失败不致使申请永久悬挂（可补发/催办）；被拒员工可重新申请；已批准/已停用状态不被重复事件重置。

### 5.4 离职与账户停用（平台核心价值）
- **F-5.4.1 事件驱动停用**：员工在身份源离职/冻结/离职 → **实时停用**其在所有应用的访问（一次操作关闭全部应用）。
- **F-5.4.2 对账兜底**：定时核对"平台记为有权"的员工在身份源的在职状态，补上漏掉的事件；核对范围须覆盖所有当前有权用户。
- **F-5.4.3 离开识别**：员工在身份源已不存在（硬删除）判为离职并停用，与"身份源接口瞬时不可用"区分（后者不误停）。
- **F-5.4.4 绝不遗漏**：停用失败须重试到成功或**触发人工介入告警**——宁可重试/告警，不可漏停。
- **F-5.4.5 分钟级全线撤权**：应用会话须能在停用后**≤5 分钟**内失效（即便员工已登录），保证离职后各应用访问及时切断。

### 5.5 授权模型（三层）
| 层 | 管什么 | 特性 |
|---|---|---|
| **① 身份** | 你是谁 | 来自身份源，全平台唯一 |
| **② 准入** | 能不能进应用 X | **应用级**，逐个授予/回收 |
| **③ 角色** | 进去能干什么 | **应用级**，含义由应用自定义 |

- **F-5.5.1 分配中心化**：谁有什么准入/角色，集中在平台治理——一处审计、离职一键全线失效。
- **F-5.5.2 含义应用自理**：角色的具体权限由应用定义，平台只声明"某员工在某应用有某角色"。
- **F-5.5.3 审批即赋角（可选）**：审批时可直接指定角色。

### 5.6 会话与访问撤销
- **F-5.6.1** 应用会话须为**可撤销**（登出即失效、停用即失效），不得长期有效到与撤销脱节。
- **F-5.6.2** 撤销以安全优先：无法确认时倾向收权，但对身份源短时不可用有容错，避免误伤全员。

### 5.7 应用接入
- **F-5.7.1** 提供**标准接入规范** + 可跑通的最小示例，接入方无需理解平台内部。
- **F-5.7.2** 接入方职责最小化：完成标准登录对接、读取准入/角色做判定、待批时进入审批流；身份、审批、离职撤权由平台兜底。
- **F-5.7.3** 强制安全基线：身份信息传递只含必要字段、传输加密、来源可验证。

### 5.8 管理与审计
- **F-5.8.1** 提供管理后台：身份源、应用、准入与角色、会话的集中管理。
- **F-5.8.2** 全流程留痕：准入/拒绝/停用/角色变更均可查询、可导出。
- **F-5.8.3**（规划）"一张表看清谁能访问什么"的统一视图 + 离职撤权核对报表。

### 5.9 通知
- **F-5.9.1** 审批请求、停用失败告警、申请结果均通过员工日常使用的沟通工具（当前飞书）触达。

## 6. 非功能需求

| 维度 | 要求 |
|---|---|
| **安全** | 认证与授权分离；准入默认拒绝（fail-closed）；**离职绝不漏禁**；密钥集中托管；身份信息最小化下发；审批回调来源可验证 |
| **可用性** | 关键路径（登录、撤权）高可用、故障可自恢复；核对/撤权任务不可因单点故障中断 |
| **性能** | 支撑千人级规模；登录与会话校验低延迟；批量核对离峰运行 |
| **可扩展** | 加身份源不改应用、加应用不改平台 |
| **合规审计** | 身份数据留公司内网；全流程留痕；数据可备份、可恢复 |
| **运维** | 配置与密钥外置；可观测（关键动作有日志）；具备备份与恢复流程 |

## 7. 分期与优先级

按能力分期交付，前一期为后一期地基（不含实现进度）：

| 期 | 能力范围 | 优先级 |
|---|---|---|
| **里程碑 1 · 单应用闭环** | 身份源登录 + SSO + 单应用入职审批 + 离职自动停用 | P0 必须 |
| **里程碑 2 · 多应用推广** | 每应用独立准入/角色/审批人，多应用共享一套身份与生命周期 | P0 必须 |
| **里程碑 3 · 平台化治理** | 统一权限视图、审批结果回告、应用内角色强制策略 | P1 |
| **里程碑 4 · 多身份源与自动供给** | 邮箱/密码及其他身份源、外部合作方、账号自动供给/回收 | P2 |

## 8. 成功指标（KPI）

| KPI | 目标 |
|---|---|
| 断权时延 | 离职后全线断权 **≤5 分钟**（抽样核对） |
| 漏停率 | **0** —— 离职员工在任何应用留有有效访问即事故 |
| 接入成本 | 新应用接入 **≤1 人日** |
| 覆盖 | 接入应用数达 **6–15** |
| 审批 SLA | 申请到批准中位时长 **< 1 工作日** |

---

## 9. 风险与依赖

| 风险 / 依赖 | 应对方向 |
|---|---|
| 身份数据丢失 | 定期备份 + 异地留存 + 恢复演练 |
| 身份源接口 / 事件依赖 | 事件 + 定时对账双通道；停用失败人工兜底告警 |
| 关键服务可用性 | 高可用部署 / 故障自恢复；核对任务不漏跑 |
| 审批人缺位导致申请悬挂 | 催办 + 可重新申请 + （规划）超时升级 |

---

## 10. 范围外

- 应用内业务权限的具体语义。
- 组织架构 / HR 主数据（以既有身份源为准）。
- 面向外部终端用户（ToC）的账号体系。
